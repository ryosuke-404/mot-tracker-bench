## 1. 評価方法

### 1.1 GTなし評価と GTあり評価の違い

本評価では「動画内に7人が映っている」という事前知識を**グラウンドトゥルース (GT)** として活用する。
フレーム単位のアノテーション（各人のBBoxを手動入力）は行わず、**人数のみをGTとして使用**する近似評価方式を採用した。

```
アノテーション不要の評価方式：
  GT情報  : 動画内の人数 = 7人
  評価対象 : 各トラッカーが生成するトラックIDの数・寿命・一致性
```

### 1.2 評価パイプライン

```
動画フレーム
    ↓
YOLOX-s 検出（高スコア / 低スコア に分類）
    ↓
OSNet-x0.25 Re-ID特徴抽出（512次元ベクトル）
    ↓
各トラッキングアルゴリズム
    ↓
Re-IDギャラリー照合・ID救済
    ↓
指標計算
```

## 2. 評価指標の定義

### 2.1 GT-based 指標（GT人数を使用）

#### GTスコア（gt_score）★ メイン指標
**IDprecision・Coverage・過検出ペナルティの調和平均**

$$\text{GT-score} = \text{HarmonicMean}\left(\text{IDprec},\ \text{Cover},\ \frac{1}{\text{ODrate}}\right)$$

- 値域: 0〜1　**高いほど良い**
- 3つの要素が **すべて高くないと** スコアが上がらない（調和平均の特性）
- 1つでも極端に悪い要素があると全体スコアが下がる

---

#### ID精度（id_precision）
**発生したトラックIDのうち、GT人数分だけが「正しい」という前提での精度**

$$\text{IDprec} = \min\left(\frac{\text{GT人数}}{\text{track\_fragments}},\ 1.0\right)$$

| 値 | 意味 |
|----|------|
| 1.0 | トラックID数 = GT人数（理想） |
| 0.5 | トラックID数 = GT人数の2倍（断片化） |
| 0.1 | トラックID数 = GT人数の10倍（非常に断片化） |

**注意**: この指標はIDブレーク（同じ人が別IDになる）が少ないほど高くなる。

---

#### カバレッジ（coverage）
**GT人数の全員が動画の50%以上のフレームで追跡されているか**

$$\text{Coverage} = \min\left(\frac{\text{長寿命トラック数}}{\text{GT人数}},\ 1.0\right)$$

- 長寿命トラック = 総フレーム数の50%以上生存したトラック
- 値域: 0〜1　**高いほど良い**（1.0 = 全員を長期追跡できた）

---

#### 過検出率（over_det_rate）
**同時に追跡されている平均人数がGT人数の何倍か**

$$\text{ODrate} = \frac{\text{平均アクティブトラック数}}{\text{GT人数}}$$

| 値 | 意味 |
|----|------|
| 1.0x | ちょうど7人を追跡（理想） |
| 2.0x | 平均14人を追跡（幽霊トラック多数） |
| 0.9x | 平均6.3人（一部見逃し） |

---

#### 断片化率（frag_rate）
**1人あたりに何個のトラックIDが生成されたか**

$$\text{FragRate} = \frac{\text{track\_fragments}}{\text{GT人数}}$$

- 理想値: 1.0x（1人1ID）
- 現実的な目標: 5x以下（1人5ID以内）

---

#### ID回収率（id_recovery）
**IDブレーク（断片化）のうち、Re-IDによって何割を回収したか**

$$\text{IDrecov} = \min\left(\frac{\text{rescue\_count}}{\text{track\_fragments} - \text{GT人数}},\ 1.0\right)$$

- 値域: 0〜1　**高いほど良い**（1.0 = 全断片をRe-IDで復元）
- Re-IDが機能していないトラッカー（SORTなど）はそもそも断片が少なく分母が小さい

---

### 2.2 プロキシ指標（GTなしで計算可能）

| 指標 | 定義 | 理想値 |
|------|------|--------|
| **avg_conf** | 高スコア検出の平均信頼度 | 高いほど良い |
| **det_per_frame** | フレームあたり平均検出数（高+低スコア） | GT人数に近いほど良い |
| **track_fragments** | 生成されたユニークトラックID総数 | 少ないほど良い |
| **avg_track_life** | 平均トラック寿命（フレーム） | 長いほど良い |
| **long_track_ratio** | 30フレーム超生存トラックの割合 | 高いほど良い |
| **rescue_count** | Re-IDレスキュー回数 | 高いほどRe-IDが機能 |
| **rescue_rate** | 新規テンタティブ中のレスキュー率（%） | 高いほど良い |
| **avg_rescue_sim** | レスキュー時の平均コサイン類似度 | 高いほど信頼性が高い |
| **avg_det_ms** | 検出推論時間（ms/frame） | 低いほど高速 |
| **total_time_s** | 総処理時間（秒） | 低いほど高速 |

## 4. トラッカー別詳細分析

### 4.1 SORT（1位、GTスコア 0.449）

```
Frags=26  FragRate=3.7x  Life=167.9フレーム  ODrate=1.97x
```

**強み**
- 断片化が全トラッカー中で圧倒的に最少（frags=26）
- 平均トラック寿命167.9フレーム（2位Hybrid-SORTの3.5倍）
- 長寿命トラック率88%（7人全員が動画の大半で同一IDを維持）

**弱み**
- ODrate=1.97x：平均13.8人を同時追跡（幽霊トラック多数）
- Re-IDレスキュー機能していない（IDrecov=0.105）
- 純粋IoU+Kalmanのみ → 遮蔽・離席からの復帰が困難

**SORTが1位になる理由**
SORTはID数が少ないため IDprec が最高値になる。ただし「幽霊トラック（存在しない人の追跡）」が多いことを ODrate=2x が示している。バスOD調査では幽霊トラックがトリップワイヤを通過すると誤カウントが発生する。

---

### 4.2 Hybrid-SORT（2位、GTスコア 0.304）

```
Frags=55  FragRate=7.9x  Life=48.3フレーム  ODrate=0.98x  IDrecov=0.625
```

**強み**
- ODrate=0.98x：最も「ちょうど7人」に近い追跡
- 高さ・形状一貫性コストにより、バス座席の高さが均一な環境に適合
- ID回収率62.5%（Re-IDが有効に機能）

**弱み**
- SORTより断片数は多い（55 vs 26）
- Re-IDなしSORTに比べIDブレークが多いが、それをRe-IDで補っている

---

### 4.3 Deep OC-SORT（3位、GTスコア 0.296）

```
Frags=57  FragRate=8.1x  Life=44.8フレーム  ODrate=0.99x  IDrecov=0.500
```

**強み**
- OC-SORT + EMA外観 + OCR再初期化の組み合わせ
- 遮蔽後の再追跡精度が高い（OCR: 最後の観測からKalmanを再初期化）
- ODrate≈1.0（幽霊トラックが少ない）

---

### 4.4 OC-SORT（4位、GTスコア 0.292）

```
Frags=58  FragRate=8.3x  IDrecov=0.686（全Re-ID系で最高）
```

**強み**
- ID回収率68.6%はRe-IDを活用するトラッカーの中で最高
- 観測中心の速度推定でKalman状態のドリフトを抑制

---

### 4.5 ByteTrack（5位、GTスコア 0.284）

```
Frags=60  FragRate=8.6x  Life=43.8フレーム  ODrate=1.00x
```

現在のシステムのデフォルトトラッカー。5位と安定した性能。  
高スコア/低スコアの2段階マッチングで遮蔽中の人物を追跡。

---

### 4.6 BoT-SORT（13位、GTスコア 0.161）

```
Frags=116  FragRate=16.6x  IDrecov=0.688
```

**なぜ下位か**
- GMC（グローバルモーション補正）はCPU・バス車内という環境では逆効果
- ORBキーポイントマッチングが不安定 → 誤ったKalman補正 → 断片化増加
- GPU環境・高fps動画（60fps以上）では効果を発揮する設計

---

### 4.7 UCMCTrack（15位、GTスコア 0.138）

```
Frags=138  FragRate=19.7x  ODrate=1.10x
```

**なぜ下位か**
- ホモグラフィ（地上面座標変換）はカメラキャリブレーションデータが必要
- デフォルトの推定ホモグラフィがこのカメラ設置角度に合っていない
- 適切なキャリブレーション後なら精度向上が期待できる

## 5. パラメータチューニング実験

### 5.1 チューニング方針

ベースライン評価で判明した2種類の問題パターンに対して、それぞれ異なる戦略でチューニングを実施した。

| 問題パターン | 代表トラッカー | 現象 | 対策 |
|---|---|---|---|
| **幽霊トラック型** | SORT | ODrate=1.97x（平均14人追跡） | `track_thresh`を上げてノイズ検出を排除 |
| **断片化型** | ByteTrack/OC-SORT等 | frags>>7だがODrate≈1.0x | `track_buffer`延長 + `match_thresh`緩和 |

### 5.2 チューニング第1回（3バリアント）

| バリアント | 変更内容 | Before | After | 改善率 |
|---|---|---|---|---|
| **SORT ★** | `track_thresh: 0.45→0.60` | 0.449 | **0.647** | **+44%** |
| BoT-SORT | `gmc_method: "orb"→"none"` | 0.161 | 0.157 | −2%（悪化） |
| SparseTrack | `n_layers: 3→1` | 0.163 | 0.163 | 変化なし |

**結果**: SORT の `track_thresh=0.60` が劇的に有効。frags 26→16、ODrate 1.97x→1.35x。  
BoT-SORT はGMC無効化でも改善せず → 根本問題はGMCではない。  
SparseTrack は層数変更の影響なし → 断片化の根本原因は別にある。

### 5.3 チューニング第2回（5バリアント）

断片化型トラッカーに対して `track_buffer=90→180`（消えたトラックをより長く保持）と  
`match_thresh=0.85→0.90`（マッチングをより積極的に）の組み合わせを適用。

| バリアント | 変更内容 | Before | After | 改善率 |
|---|---|---|---|---|
| **ByteTrack ★** | buffer=180, match=0.90 | 0.284 | **0.368** | **+30%** |
| **OC-SORT ★** | buffer=180, match=0.90, delta_t=5 | 0.292 | **0.375** | **+28%** |
| **Hybrid-SORT ★** | buffer=180, match=0.90 | 0.304 | **0.362** | **+19%** |
| **Deep OC-SORT ★** | buffer=180, match=0.90, reid=0.55 | 0.296 | **0.382** | **+29%** |
| **TransTrack ★** | buffer=180, appear=0.60 | 0.262 | **0.362** | **+38%** |

**結果**: 全5バリアントで改善。`track_buffer` 延長が最も効果的（消えた人物を長く記憶することで再登場時に既存IDに紐付けられる）。

### 5.4 チューニング効果のまとめ

```
SORT tuned (0.647)                    ★ 全体最高
├── SORTの幽霊トラック問題を解決
└── track_thresh=0.60 が鍵

Deep OC-SORT tuned (0.382) / OC-SORT tuned (0.375)
├── バッファ延長で frags 57→41 に削減
└── ODrate=0.96x（若干の過少検出が発生）

ByteTrack tuned (0.368) / Hybrid-SORT tuned (0.362)
└── 同様の改善傾向
```

## 6. 評価指標の限界と注意事項

### 6.1 「Coverage=1.0」の解釈

**全トラッカーでCoverage=1.000** になっている。これは：
- 7人全員が動画の50%以上のフレームに映っているため、長寿命トラックが GT ≥ 7 個存在
- SORTのように幽霊トラックが多くても「長寿命なら長寿命としてカウント」される

Coverage は「追跡の継続性」を測るが、**正しい人物を追跡しているか** は判定できない。

### 6.2 IDprec の低さ（最高でも 0.438）

全トラッカーで IDprec が低いのは、**断片化が多いから**ではなく、  
バス車内で7人が長時間映り続けるため、離席・遮蔽・再登場のたびに新IDが発生するため。  
これはトラッカーの問題というより **Re-IDギャラリーの閾値設定** の問題でもある。

### 6.3 SORT tuned の「総合1位」についての注意

SORT tuned (0.647) は全バリアント中最高スコアだが、ODrate=1.35x（平均9.5人追跡）は  
チューニング前より改善したものの **まだ幽霊トラックが残っている**。  
バスOD乗降カウントでは、幽霊トラックがトリップワイヤを通過すると **誤検知** につながる。  
真に幽霊ゼロを重視する場合は Deep OC-SORT tuned (ODrate=0.96x) の方が安全な選択。

### 6.4 本評価の近似誤差

| 要因 | 影響 |
|------|------|
| フレーム単位のBBoxアノテーションなし | MOTA/IDF1/HOTAは計算不可 |
| 人数のみをGTとして使用 | Coverage・IDprecが近似値 |
| 動画の一部（300フレーム）のみ | 乗降イベントが発生していない可能性あり |
| CPU環境での評価 | GPU環境では速度・精度比が変わる |

## 7. バスOD調査システムへの推奨

### 7.1 用途別推薦（チューニング結果反映）

| 優先事項 | 推奨（tuned版） | GTスコア | 理由 |
|---------|------|------|------|
| **総合スコア最高** | **SORT tuned** | **0.647** | 断片数最少・IDprec最高 |
| **乗降カウント精度** | **Deep OC-SORT tuned** | 0.382 | ODrate=0.96x（幽霊ほぼゼロ）|
| **ID回収率最高** | **Hybrid-SORT tuned** | 0.362 | IDrecov=0.730（Re-ID最活用） |
| **最速（リアルタイム）** | **GHOST** | 0.227 | 130.8秒（他の半分以下） |
| **GPU本番環境** | **BoT-SORT** | 0.161 | GMCはGPU+高fpsで真価発揮 |

### 7.2 最終推奨：SORT tuned（現在のデフォルト）

バスOD調査の目的（乗降人数の正確なカウント）に対して：

```
✅ GTスコア 0.647  → 全18バリアント中1位
✅ Frags=16        → 7人に対し最少断片数（理想は7）
✅ Life=179.9フレーム → 全バリアント中最長（継続追跡が安定）
✅ Coverage=1.0    → 7人全員を長期追跡
⚠ ODrate=1.35x   → 幽霊トラックがまだ残存（9.5人相当を追跡）
```

幽霊トラックゼロを最優先する場合は **Deep OC-SORT tuned**（ODrate=0.96x）を推奨。

> **現在のデフォルト設定**: `config/system.yaml` の tracker は SORT tuned に更新済み。  
> `run_bus.py` / `run_video.py` も SORTTracker (track_thresh=0.60) を使用。

## 8. 評価スクリプトの実行方法

```bash
# GT人数を指定して全15トラッカーを評価
.venv/bin/python scripts/evaluate.py \
    --video      assets/142755-780943401_medium.mp4 \
    --frames     300 \
    --skip       2 \
    --scale      0.4 \
    --device     cpu \
    --gt_persons 7 \
    --out        outputs/eval_gt7.json

# 特定のトラッカーのみ評価
.venv/bin/python scripts/evaluate.py \
    --gt_persons 7 \
    --trackers   hybridsort ocsort bytetrack sort

# RTDETRv2 + ByteTrack の比較
.venv/bin/python scripts/evaluate.py \
    --gt_persons 7 \
    --detectors  rtdetr yolox_s \
    --trackers   bytetrack hybridsort
```
